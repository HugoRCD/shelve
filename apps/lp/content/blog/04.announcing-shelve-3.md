I meant to write this when Shelve 3.0 shipped. Life happened, the project kept moving, and here we are at **3.2** with **CLI 5.1**. This post catches up on everything that changed since **2.3.3** — the release where Shelve stopped feeling like a polished side project and started behaving like infrastructure you can trust with production secrets.

If you have been using Shelve since [2.0](/blog/announcing-2.0), you already know the pitch: centralize your environment variables, inject them at runtime with `shelve run`, sync to GitHub Secrets, share securely with Vault. Version 2 was about polish — a new visual identity, a better CLI, and a workflow that finally made `.env` files optional on your laptop.

Version 3 is different. It is not about new screens. It is about rebuilding every layer of the stack — how tokens are generated, how secrets are encrypted, how the CLI authenticates, how actions are audited — to the standard you would expect from a tool hosting your production API keys. And it is the first release explicitly designed for the way many of us build software now: surrounded by AI coding agents that read every file in the workspace and quietly turn `.env` into prompt context.

## Before you upgrade

::callout{type="important"}
**Breaking changes in 3.0.** Plan a short maintenance window before upgrading production or CI.
::

- **API tokens were invalidated by the migration.** Tokens are now generated with `crypto.randomBytes(32)` and stored as SHA-256 hashes. Plaintext is returned only at creation. Re-issue every token after upgrading.
- **The CLI now sends `Authorization: Bearer <token>`.** Cookie-based auth still works for one release window with `Deprecation` and `Sunset` response headers, but Bearer is the default going forward.
- **Upgrade the CLI:** `npm i -g @shelve/cli@latest` (or your package manager equivalent).

See the [CLI upgrade guide](/docs/cli/upgrade) and [API tokens docs](/docs/core-features/tokens) for details.

## Security foundation (v3.0 / CLI 5.0)

### Hashed, scoped API tokens

API tokens used to be generated with `Math.random()` and stored reversibly in the database. That was fine for a side project. It is not fine for something hosting your production secrets.

Now:

- Tokens are **256 bits of entropy**, encoded in [Crockford base32](https://www.crockford.com/base32.html).
- Only **`sha256(token)`** and a non-secret **prefix** (`she_…`) are stored. The plaintext is shown **exactly once** at creation.
- Lookup is an **O(1) hash query** with `timingSafeEqual` — no decrypting every token in a loop.
- Tokens carry **granular scopes**: `teamIds`, `projectIds`, `environmentIds`, and `permissions: ('read' | 'write')[]`, enforced server-side via `requireTokenScope`.
- Optional **`expiresAt`**, **CIDR allowlists**, and **`lastUsedAt` / `lastUsedIp`** tracking on every authenticated request.

The token UI at `/user/tokens` shows the prefix instead of the full secret, displays scopes and expiry, and reveals the value only once at creation. In **3.1**, the creation flow moved to a roomier modal with cascading multi-select pickers for teams, projects, and environments, inline CIDR validation, and a clear warning when a token is fully unscoped.

### Bearer authentication

The CLI used to authenticate via `Cookie: authToken=…`. It worked, but it was the wrong default for bots, agents, CI, and integrations.

CLI 5.0 sends `Authorization: Bearer she_…`. The server prefers Bearer and falls back to cookies with deprecation headers. Cleaner, more standard, easier to wire into everything else in your stack.

### Envelope encryption, per project

Until 3.0, every variable in the database was encrypted with a single global key. One leak, one big problem.

Shelve now uses **envelope encryption**:

```
value ──seal──▶ ciphertext     (key = DEK, per-project)
DEK   ──seal──▶ encryptedDek   (key = KEK, platform-wide)
```

Each project owns a **Data Encryption Key (DEK)** — a random 256-bit key generated on first write, sealed with the platform **Key Encryption Key (KEK)**, and persisted alongside the project. Variables are encrypted with the project DEK. Rotating a project's key only re-encrypts that project's variables. A leaked DEK scopes the blast radius to a single project.

Projects created before the upgrade keep working: decryption tries the project DEK first, then transparently falls back to the KEK. No data migration required. Read the full scheme in the [encryption docs](/docs/core-features/encryption).

### Audit logs

Every security-relevant action — token create/delete, variable read/write, environment and project mutations — is recorded in a new **`audit_logs`** table with actor type/id, action, resource, IP, user agent, and arbitrary metadata. The `logAudit` helper swallows errors so logging never blocks the main request.

Team admins get a new **Audit logs** tab in team settings with cursor pagination and action filters. In **3.1**, the UI got color-coded action badges, resource icons, parsed user-agent labels (e.g. `Shelve CLI 5.0.0`, `Chrome · macOS`), and a per-row metadata popover to inspect the full JSON payload.

Query the feed via `GET /api/teams/[slug]/audit-logs`. See [audit logs docs](/docs/core-features/audit-logs).

## The CLI as a runtime layer (CLI 5.0)

`shelve run` was introduced in 2.0 as a way to skip the `.env` file. In 5.0 it became a **true runtime layer** — the default path for local development, CI, and agent shells.

### OS keychain storage

On your machine, the CLI used to drop a plaintext JSON file in `~/.shelve`. Now tokens live in the **OS keychain** via `@napi-rs/keyring` (macOS Keychain, Windows Credential Manager, libsecret), with a hardened `0600` fallback at the XDG path (`~/.config/.shelve`). The legacy file is migrated automatically on first read. `shelve logout` clears both the keychain entry and the rc file.

### Offline cache, watch mode, and secret references

After every successful pull, `shelve run` writes an **AES-256-GCM cache** keyed from your API token via HKDF. Revoking the token makes the cache unreadable. If the API call fails, `shelve run` transparently falls back to a fresh-enough cache and prints a warning instead of crashing.

New capabilities:

- **`--offline`** — force cache, fail if absent
- **`--no-cache`** — never read or write cache
- **`--cache-ttl 24h`** — override freshness
- **`--watch`** — poll Shelve for variable changes and forward `SIGHUP` to the child (Vite/Nuxt/Next handle the reload)
- **`--restart-on-change`** — kill and respawn instead
- **`--template .env.template`** — resolve `shelve://<team>/<project>/<env>/<KEY>` references against live or cached secrets; the template file can be safely committed to Git

Signal forwarding, watch mode, and cross-platform spawn were hardened in **5.0.3** and **5.1.0** — including Windows `.cmd` shim support, parent-death watchdogs, and fixes for orphaned child processes that had been lurking since the early `shelve run` days.

```bash
shelve init
shelve run -- pnpm dev
shelve run --watch -- pnpm dev
shelve run --template .env.template -- pnpm build
```

### AI-agent guards

This is where 3.x diverges most from the 2.x story. AI coding agents (Cursor, Claude Code, Codex, Aider, Continue, and more) can read every file in your workspace unless you tell them otherwise. If you run `shelve pull` and write a `.env` to disk, an agent can trivially exfiltrate its contents into a prompt or a suggested commit.

Shelve 3 makes that workflow safer by default — not by adding warnings, but by making the plaintext file on disk the exception:

- **`shelve init`** writes `.cursorignore`, `.aiderignore`, `.codeiumignore`, `.continueignore`, and a `# shelve-managed-block` in `.gitignore`.
- **`shelve pull`** detects AI-agent environments and **prompts before writing plaintext** (skip with `--yes`). In agent shells without `--yes`, it fails immediately with `AGENT_BLOCKED`.
- **`shelve run`** keeps secrets in the child process memory only — never on disk.

Prefer `shelve run -- <cmd>` over `shelve pull`. Always.

## Platform polish (v3.0 → v3.1)

Between the security foundation and the agent automation wave, a lot of platform work landed:

- **Variable groups and descriptions** — organize variables in the UI; `pull` outputs section headers (`# ---- Group ----`) and inline description comments, plus auto-generates a `.env.example`.
- **OTP auth** and **bot-id protection** for the login surface.
- **Documentation migrated to Docus** — clearer structure, better search, CLI reference on shelve.cloud.
- **Vault made app-agnostic** — deployable independently of the main Shelve app.
- **Vercel self-hosting docs** and updated terms/privacy pages.
- **Test infrastructure** — unit and E2E coverage for push/pull, tokens, bearer auth, audit logs, offline cache, credentials, and secret references.
- **Navigation and members table** refactors, single-team auto-redirect, Nuxt 4 and Nuxt UI upgrades.

None of these are as flashy as envelope encryption, but they matter if you run Shelve daily with a team.

## Built for agents and teams (v3.2 / CLI 5.1)

The latest wave makes Shelve a first-class citizen in CI pipelines and agent shells.

### Global flags and structured output

Every command now supports agent-friendly global flags:

| Flag | Purpose |
|------|---------|
| `--json` | Machine-readable success on stdout, structured errors on stderr |
| `--quiet` / `-q` | Suppress spinners and intro/outro |
| `--yes` / `-y` | Skip confirmation prompts |
| `--non-interactive` | Fail fast instead of prompting (auto-enabled in CI and agent shells) |
| `--debug` | Verbose logs without Authorization headers or secret values |

New **`shelve doctor`** validates your setup and reports health checks. **`shelve login --token`** accepts a token non-interactively. Query commands return JSON shapes documented in [Agents & automation](/docs/cli/agents-automation).

### Sync policies, diff, and protected environments

Push and pull used to be optimistic: last write wins, with little guard against accidentally overwriting production from a laptop.

**3.2** introduces **sync policies** in `shelve.json` and on the server per project:

- **`shelve diff`** — compare local `.env` with Shelve (no writes, safe with `--json`)
- **`shelve sync`** — reconcile according to `sourceOfTruth` (`remote` vs `local`)
- **`onPushConflict`** — `overwrite`, `skip`, `fail`, or `prompt`
- **`pullMode`** — `replace` or `merge` (local-only keys kept)
- **`protectedEnvironments`** — server-side `allowPush: false`; API writes rejected with `ENV_PROTECTED`

Server policies cannot be relaxed from the CLI. If production is protected on the server, your laptop cannot override it. See [sync policies docs](/docs/cli/sync-policies).

### Agent skill and CI tooling

Install the published agent skill so Cursor, Claude, and other agents know how to use Shelve safely:

```bash
npx skills add https://shelve.cloud
```

The skill covers `shelve run`, `shelve init`, scoped tokens, sync policies, and the `--json` / `--non-interactive` flags. A **GitHub Action** is available for CI. Full CLI documentation, troubleshooting guides, and `llms.txt` skill links live on [shelve.cloud/docs/cli](/docs/cli).

## What this means for you

| Persona | What changed |
|---------|--------------|
| **Solo developer** | Run `shelve init` once, then `shelve run -- pnpm dev`. Secrets stay off disk and out of agent context. Offline cache keeps you unblocked when the API hiccups. |
| **Team admin** | Issue scoped, expiring tokens with IP allowlists. Review the audit log. Protect production environments server-side. Envelope encryption limits blast radius per project. |
| **CI / agent automation** | `SHELVE_TOKEN` + `--json` + `--non-interactive`. Use `shelve doctor --json` in setup steps. Prefer `shelve run` over `pull`. Install the agent skill. |

Shelve 2.x made environment management pleasant. Shelve 3.x makes it **trustworthy** — and safe in a world where your IDE reads every file you have.
