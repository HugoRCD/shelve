I meant to write this when Shelve 3.0 shipped. Life happened, the project kept moving, and here we are at **3.2** with **CLI 5.1**. This post catches up on everything that changed since **2.3.3** — a big step forward for anyone running Shelve in a team, in CI, or alongside AI coding agents.

If you have been using Shelve since [2.0](/blog/announcing-2.0), you already know the pitch: centralize your environment variables, inject them at runtime with `shelve run`, sync to GitHub Secrets, share securely with Vault. Version 2 delivered on that promise — a polished UI, a capable CLI, and a workflow that made `.env` files optional on your laptop.

Version 3 builds on that foundation. The theme is **more control, more visibility, and a CLI that keeps up with how we actually work today** — especially when AI agents share your workspace. Here is what landed, release by release.

## What's new at a glance

::callout{type="tip"}
Already on Shelve 2.x? Your secrets, projects, and workflows carry over. Most of 3.x is additive — new capabilities on top of what you already use.
::

| Area | Highlights |
|------|------------|
| **Tokens** | Scoped permissions, expiry, IP allowlists, usage tracking |
| **Encryption** | Per-project envelope encryption on top of existing AES-256 |
| **Observability** | Full audit log with a polished team UI |
| **`shelve run`** | Offline cache, watch mode, `shelve://` templates, OS keychain |
| **Agents & CI** | `shelve init`, `--json`, `shelve doctor`, sync policies, agent skill |
| **Platform** | Variable groups, OTP auth, Docus docs, E2E test coverage |

## Before you upgrade

::callout{type="important"}
**A few breaking changes in 3.0.** Plan a short maintenance window before upgrading production or CI.
::

- **API tokens need to be re-issued after upgrading.** The token format and storage model changed — plaintext is now shown only at creation, and tokens are stored as hashes. Create new tokens in the UI and update your CI secrets.
- **The CLI now sends `Authorization: Bearer <token>`.** Cookie-based auth still works for one release window with `Deprecation` and `Sunset` response headers.
- **Upgrade the CLI:** `npm i -g @shelve/cli@latest` (or your package manager equivalent).

See the [CLI upgrade guide](/docs/cli/upgrade) and [API tokens docs](/docs/core-features/tokens) for details.

## More control over API tokens (v3.0 / CLI 5.0)

Shelve has always encrypted secrets at rest. With 3.0, **API tokens get the same level of care** — plus a set of features teams have been asking for:

- **256 bits of entropy**, encoded in [Crockford base32](https://www.crockford.com/base32.html).
- **Hashed at rest** — only `sha256(token)` and a display **prefix** (`she_…`) are stored. The full value is shown **exactly once** at creation.
- **Granular scopes**: restrict a token to specific teams, projects, environments, and `read` / `write` permissions — enforced server-side via `requireTokenScope`.
- **Optional expiry**, **CIDR allowlists**, and **`lastUsedAt` / `lastUsedIp`** on every request so you always know which token did what.

The token UI at `/user/tokens` shows scopes, expiry, and last-used info at a glance. In **3.1**, the creation flow moved to a roomier modal with cascading multi-select pickers, inline CIDR validation, and a helpful nudge when a token has no restrictions applied.

### Standard Bearer authentication

The CLI now sends `Authorization: Bearer she_…` — the same pattern used by most APIs and CI systems. Cookie-based auth remains available during the transition, with clear deprecation headers. If you integrate Shelve with scripts, GitHub Actions, or agent tooling, Bearer makes the wiring straightforward.

### Envelope encryption, per project

Shelve already sealed every variable with **AES-256-GCM** before it hit the database. Version 3 adds **envelope encryption** — an extra layer that gives you finer-grained key management:

```
value ──seal──▶ ciphertext     (key = DEK, per-project)
DEK   ──seal──▶ encryptedDek   (key = KEK, platform-wide)
```

Each project gets its own **Data Encryption Key (DEK)**, sealed by the platform **Key Encryption Key (KEK)**. That means per-project key rotation without touching application code, and a smaller blast radius if a single project key is ever compromised.

Existing projects upgrade seamlessly: decryption tries the project DEK first, then transparently falls back to the previous scheme. No data migration required. Read the full scheme in the [encryption docs](/docs/core-features/encryption).

### Audit logs — finally

One of the most requested features: a full **audit trail** for your team.

Every sensitive action — token create/delete, variable read/write, environment and project mutations — is recorded with actor, action, resource, IP, user agent, and metadata. Team admins get an **Audit logs** tab in team settings with cursor pagination and action filters.

In **3.1**, the UI got color-coded action badges, resource icons, parsed user-agent labels (e.g. `Shelve CLI 5.0.0`, `Chrome · macOS`), and a per-row metadata popover to inspect the full JSON payload. Query the feed via API too — see [audit logs docs](/docs/core-features/audit-logs).

## `shelve run` grows up (CLI 5.0)

`shelve run` was the headline feature of [2.0](/blog/announcing-2.0): run any script with secrets injected at runtime, no `.env` on disk. In CLI 5.0 it became significantly more capable — and reliable.

### Credentials in the OS keychain

Your CLI token now lives in the **OS keychain** (macOS Keychain, Windows Credential Manager, libsecret), with a hardened XDG fallback at `~/.config/.shelve`. If you had credentials in the legacy `~/.shelve` file, they migrate automatically on first run.

### Offline cache, watch mode, and secret references

Working on a plane? API briefly down? After every successful fetch, `shelve run` keeps an **encrypted offline cache** (AES-256-GCM, keyed from your token via HKDF). Revoke the token and the cache becomes unreadable.

New flags and modes:

- **`--offline`** — use cache only
- **`--no-cache`** — always fetch fresh
- **`--cache-ttl 24h`** — control freshness
- **`--watch`** — poll for variable changes and forward `SIGHUP` to the child (Vite/Nuxt/Next hot-reload)
- **`--restart-on-change`** — respawn the process instead
- **`--template .env.template`** — resolve `shelve://<team>/<project>/<env>/<KEY>` references; commit the template safely to Git

Signal forwarding, watch mode, and cross-platform spawn were also hardened in **5.0.3** and **5.1.0** — Windows `.cmd` shims, parent-death watchdogs, and cleaner exit-code propagation.

```bash
shelve init
shelve run -- pnpm dev
shelve run --watch -- pnpm dev
shelve run --template .env.template -- pnpm build
```

### Built for the AI-agent era

AI coding agents (Cursor, Claude Code, Codex, Aider, Continue, and more) are part of many workflows now. Shelve 3 meets that reality with practical guardrails:

- **`shelve init`** — one command to add `.cursorignore`, `.aiderignore`, `.codeiumignore`, `.continueignore`, and a `# shelve-managed-block` in `.gitignore`.
- **`shelve pull`** — detects agent environments and asks before writing plaintext to disk (`--yes` to skip).
- **`shelve run`** — secrets stay in the child process memory, never on disk.

If you use agents, `shelve run -- <cmd>` is still the recommended path. If you prefer `pull`, the CLI now has your back.

## Platform improvements (v3.0 → v3.1)

Alongside the security and CLI work, the platform itself got a lot of love:

- **Variable groups and descriptions** — organize variables in the UI; `pull` outputs section headers (`# ---- Group ----`) and inline description comments, plus auto-generates a `.env.example`.
- **OTP auth** and **bot-id protection** on the login surface.
- **Documentation migrated to Docus** — clearer structure, better search, full CLI reference on shelve.cloud.
- **Vault made app-agnostic** — deployable independently of the main Shelve app.
- **Vercel self-hosting docs** and updated terms/privacy pages.
- **Test infrastructure** — unit and E2E coverage across push/pull, tokens, auth, audit logs, offline cache, and secret references.
- **Navigation and members table** refactors, single-team auto-redirect, Nuxt 4 and Nuxt UI upgrades.

## Sync policies and agent automation (v3.2 / CLI 5.1)

The latest release focuses on **teams and automation**.

### Global flags and structured output

Every command now supports flags built for CI and agent shells:

| Flag | Purpose |
|------|---------|
| `--json` | Machine-readable success on stdout, structured errors on stderr |
| `--quiet` / `-q` | Suppress spinners and intro/outro |
| `--yes` / `-y` | Skip confirmation prompts |
| `--non-interactive` | Fail fast instead of prompting (auto-enabled in CI and agent shells) |
| `--debug` | Verbose logs without Authorization headers or secret values |

New **`shelve doctor`** validates your setup and reports health checks. **`shelve login --token`** accepts a token non-interactively. See [Agents & automation](/docs/cli/agents-automation) for JSON output shapes per command.

### Sync policies, diff, and protected environments

**3.2** adds **sync policies** — configurable rules for when local and remote diverge:

- **`shelve diff`** — compare local `.env` with Shelve (no writes, safe with `--json`)
- **`shelve sync`** — reconcile according to `sourceOfTruth` (`remote` vs `local`)
- **`onPushConflict`** — `overwrite`, `skip`, `fail`, or `prompt`
- **`pullMode`** — `replace` or `merge` (local-only keys kept)
- **`protectedEnvironments`** — mark production (or any env) as push-protected on the server

Server policies are the source of truth: if production is protected, the CLI respects that — no accidental overwrites from a laptop. See [sync policies docs](/docs/cli/sync-policies).

### Agent skill and CI tooling

Install the published agent skill so Cursor, Claude, and other agents know how to use Shelve:

```bash
npx skills add https://shelve.cloud
```

The skill covers `shelve run`, `shelve init`, scoped tokens, sync policies, and the `--json` / `--non-interactive` flags. A **GitHub Action** is available for CI. Full CLI documentation and troubleshooting guides live on [shelve.cloud/docs/cli](/docs/cli).

## What this means for you

| Persona | What you get |
|---------|--------------|
| **Solo developer** | Run `shelve init` once, then `shelve run -- pnpm dev`. Offline cache when you need it. Agent-safe by default. |
| **Team admin** | Scoped, expiring tokens with IP allowlists. A full audit log. Protected environments. Per-project encryption keys. |
| **CI / agent automation** | `SHELVE_TOKEN` + `--json` + `--non-interactive`. `shelve doctor --json` in your setup step. The agent skill for Cursor and friends. |

Shelve 2.x made environment management a pleasure. Shelve 3.x adds the **depth** — observability, fine-grained access control, and a CLI that fits how teams and agents work in 2026. If you are already on Shelve, upgrading unlocks a lot without changing the workflow you know.
